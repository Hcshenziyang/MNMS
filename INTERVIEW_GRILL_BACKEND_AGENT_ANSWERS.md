# 后端 + AI Agent 高压题参考答案（对应 INTERVIEW_GRILL_BACKEND_AGENT.md）

> 说明：以下答案是“面试可落地版本”，每题控制在可口述的长度。

---

## A. 项目本身

1. 分层是为了把 HTTP、业务编排、通用能力解耦。3 个月后如果全写在 View：复用差、测试难、需求变更会牵一发动全身。
2. Skill 负责业务流程，Tool 负责原子能力。例子：关键词抽取最初写在 JD Skill，后来抽到 `text_tools` 给简历优化复用。
3. 新增能力只改：新增 Skill、Agent 路由、API serializer/view/url。核心模型和已有 Skill 不应改。
4. 不是过度设计：三个流程复用同一 LLM/RAG/缓存能力，重复逻辑明显减少，故障定位也更快。
5. 当前主流程是串行，因为后一步依赖前一步上下文。并行会带来会话写入顺序错乱和状态竞争。
6. 用事务包裹 DB 写入，且只在拿到可用结果后写 assistant 消息。LLM 失败走降级文本，避免半成功状态。
7. 现在 `POST` 默认非幂等，重放会新增消息。生产要加 `Idempotency-Key` + 去重表。
8. 最像“伪 AI”的点是离线回退。反驳点：系统有 RAG 检索、会话状态、结构化元数据和持久化闭环，不是单轮套壳。
9. 闭环终点是可复用结构化数据：`AnalysisResult.structured_data` 和 `ChatMessage.metadata`。
10. 有效性可量化：完成率、平均轮次、用户复盘采纳率、关键词覆盖提升。当前代码主要完成流程，指标需补监控层。
11. `content` 给人读，`metadata/session_id` 给机器继续流程或可视化。
12. 用 JSONField 是为了演进快，牺牲了强约束和复杂查询性能。
13. topic 用字符串风险是脏数据和统计困难。迁移方案：新增 topic 维表 + 回填 + 双写过渡。
14. 会话删除用级联删消息；再做 Redis key 清理和索引版本引用清理，避免残留缓存。
15. 升级路径：JWT + refresh token + RBAC + 请求级用户隔离校验。
16. 最脆弱点是外部依赖（LLM/embedding/启动期索引）。线上先看 Tool 层和依赖健康。
17. 两种模式结果一致性不同，需在 metadata 标记 `mode/model_version`，并在分析报表中分开统计。
18. 参数错/资源不存在/权限问题走 4xx；依赖故障、未捕获异常走 5xx。
19. 反击“套壳”要讲：分层架构、状态持久化、RAG 管道、缓存策略、降级策略、容器化交付。
20. 时序：Nginx -> DRF 校验 -> 查/建会话 -> 写 user message -> RAG/LLM -> 写 assistant -> 缓存最近消息 -> 返回。

## B. Django / DRF

21. 顺序是字段反序列化与字段校验 -> `validate_<field>` -> `validate()`。
22. 选 APIView 因为接口是任务型而非标准 CRUD，手写更直观、可控。
23. MySQL 默认 RR 下仍可能有丢失更新（业务层没锁）。该项目多并发同会话写入时要额外控制。
24. `atomic` 放在一组必须一致的 DB 写操作周围，不要把长网络调用整段包住太久。
25. `select_related` 是 SQL JOIN（单值关系）；`prefetch_related` 是额外查询再在 Python 侧组装（多值关系）。
26. 看到循环里访问关联字段就要警惕 N+1；配合 query 计数或 debug toolbar 快速确认。
27. 常用起点 `2*CPU+1`，再按内存和 p95 延迟调。worker 太多会增加上下文切换和内存占用。
28. 多进程下进程内缓存不一致，所以跨请求共享数据应放 Redis。
29. 环境变量符合 12-factor；通过 `.env.example`、启动校验和 CI 检查防配置漂移。
30. 用 `/api/v1` 命名空间；v2 发布时并行保留 v1 一段时间，做兼容迁移。
31. 应加 trace_id 中间件并透传到日志、LLM 调用日志和响应头。
32. DRF 可用 Anon/User/ScopedRateThrottle；面试接口应做用户级 + IP 级限流。
33. CSRF 保护 cookie 场景，CORS 是浏览器跨域策略，认证解决“你是谁”。三者不是一回事。
34. MySQL 对 JSON 常用 generated column + BTREE 索引；没索引会导致全表扫描。
35. 先看指标定位瓶颈层，再进日志，最后下钻 SQL/外部 API。
36. 长任务不用 Celery 会占 worker，导致超时和吞吐下降。
37. 启动加载：首请求快但冷启动慢；懒加载：启动快但首请求抖。要按 SLA 选。
38. Django migration 天生有版本记录；脚本里避免不可逆和环境依赖可提升可重复性。
39. `ALLOWED_HOSTS='*'` 线上不建议，会扩大 Host Header 攻击面。
40. 先改三处：ASGI 运行栈、支持流式响应接口、Nginx 与客户端的流式配置。

## C. MySQL / Redis / 一致性

41. 大规模消息建议按 `session_id` 或 `user_id` 分片，并按时间分区做冷热分离。
42. 关键索引是 `(session_id, created_at)`；索引不是越多越好，会拖慢写入。
43. 用 `EXPLAIN ANALYZE` + 慢日志 + 真实压测验证收益，不只看“用到索引”字样。
44. RR 到 RC：锁冲突少了，但可重复读语义变弱，读到的快照行为会变化。
45. 同会话并发写要加序号或锁（如 `select for update` / 乐观锁版本号）。
46. TTL 依据数据波动频率与重算成本，不是固定拍脑袋。
47. 热点 miss 用单飞锁（singleflight）+ 短期互斥，避免 1000 并发同时回源。
48. 本项目适合 cache-aside；更新时可配合延迟双删降低脏读窗口。
49. prompt hash 风险：空白差异、模型参数差异导致误判；key 必须包含 model/temperature/version。
50. 多数场景选 AOF everysec + RDB，兼顾恢复速度和数据丢失窗口。
51. TTL 是基础，还应在会话结束或删除时主动失效相关 key。
52. Redis 宕机应退化到 DB + 本地兜底，功能可用但延迟升高。
53. 抖动时读请求可短重试，写请求要谨慎重试并保证幂等。
54. 连接池按并发请求数、SQL 时延和 DB 最大连接倒推，再压测调优。
55. 选 FAISS 是因为题库小、单机部署快；向量库适合多租户和高并发扩展场景。
56. 热切换做法：版本化索引文件，后台构建完成后原子切换指针。
57. 在索引元数据里存版本号，与 DB 配置表校验一致后再加载。
58. 低频更新可全量重建；高频更新做增量 + 周期性全量校准。
59. 先查 embedding 版本、索引更新时间、缓存命中和 query 预处理是否变化。
60. 备份要覆盖 MySQL（全量+binlog）和索引文件；明确 RPO/RTO 并定期演练恢复。

## D. AI Agent / RAG / LLM

61. Agent 状态包括当前阶段、问题索引、历史摘要。DB 存真相，Redis 存热态。
62. 当前是显式路由，优点是可控、便宜、可审计。
63. 给 prompt 编号（prompt_id），日志记录 prompt_id + model_version，结果波动可追溯。
64. 评估组合：离线 Recall@K + 人工质量打分，二者都要。
65. 防语义漂移：query 标准化、topic 过滤、重排和阈值裁剪。
66. K 小召回不足，K 大噪声和延迟上升。要通过离线评测找最优点。
67. 题库要做去重（文本哈希）和冲突标注（优先级/版本）。
68. embedding 升级通常需重建索引；可双索引灰度后切换。
69. 防注入做法：系统指令与用户输入隔离、敏感工具白名单、输出约束。
70. 用 schema 校验 + 解析容错，脏 JSON 回退到安全文本模板。
71. 抗幻觉靠检索约束、要求引用来源、规则后处理。
72. 选外部 API 是运维成本低、迭代快；代价是延迟抖动和数据治理压力。
73. 回退不能掩盖故障，要打监控指标区分“真实模型输出”和“fallback 输出”。
74. 可重复性通过记录输入、prompt、模型版本、参数实现“可解释复现”，不是逐字一致。
75. 成本控制：用户配额、token 上限、请求限流、缓存命中率优化。
76. 路由策略按任务类型 + SLA + 单次成本 + 最近失败率。
77. 日志要脱敏：PII 掩码、只存摘要或 hash，原文可选加密短期保存。
78. 供应商故障时目标是降级可用：规则化输出 + 缓存结果 + 告警。
79. 上下文过长就做记忆压缩：窗口化历史 + 分层摘要。
80. 面试评分可拆维度：准确性、结构化表达、深度、工程权衡，结合人工标注校准。
81. 当前未做 rerank，代价是召回后精度可能不足；后续可加轻量重排模型。
82. 证明方式：同一测试集对比纯 LLM 与 RAG 的正确率/一致性/满意度。
83. 离线评测集来自脱敏历史样本，严格 train/eval 分离并定期更新。
84. 风险最高输入点是用户自由文本（简历、JD、回答），可能包含注入与敏感内容。
85. 生产应加输出审查（策略规则+模型审核），当前版本仅做基础兜底。

## E. 系统设计 / 运维 / 故障

86. QPS 暴涨先扩后端与缓存层，再看 LLM 调用瓶颈，不是盲目全层扩容。
87. 关键 SLI：成功率、p95 延迟、LLM 调用失败率、RAG 命中率。
88. Nginx/Gunicorn 超时要覆盖 LLM 尾延迟，但不能无限大；按 p99 + buffer 设定。
89. 索引更新应异步构建新版本，切换时原子替换，避免线上阻塞。
90. 502 处置：1 分钟看 upstream 健康；5 分钟定位是应用崩溃还是网络；30 分钟完成回滚/降级并复盘。
91. 优先看指标定位层级，再用日志做根因分析，二者结合最快。
92. 灰度按流量比例放量，质量指标下降立即回滚到上一镜像。
93. 防成本攻击：认证、限流、配额、验证码、异常行为风控。
94. 权限模型：Candidate/HR/Admin 三角色 + 行级权限过滤。
95. 多租户要加 `tenant_id` 全链路隔离（DB、缓存、索引、日志）。
96. 生产密钥放 KMS/Vault，避免明文 `.env` 落盘和仓库泄漏。
97. `depends_on` 仅保证启动顺序，不保证可用；健康检查和等待脚本是必须的。
98. 镜像优化：多阶段构建、精简基础镜像、依赖分层缓存、清理无用包。
99. 复盘必备：影响范围、时间线、根因、处置、长期改进与 owner。
100. 重做一版：保留分层与缓存，重构认证与异步化，补齐观测和测试体系。

## F. 快问快答

101. 幂等是重复请求结果一致；本项目 `POST /jd/analyze` 非幂等。
102. 因为它创建新消息和副作用，不是覆盖同一资源，语义应是 POST。
103. 不能，数据库事务无法覆盖外部 HTTP 调用。
104. Redis 在本项目是缓存和状态加速层，不是最终数据真相。
105. `related_name='messages'` 让反向查询更清晰、代码可读性更好。
106. 不等价，ORM 慢可能是 N+1/序列化开销，SQL 本身可能并不慢。
107. 缓存命中高仍慢，常见是 LLM 外部调用、序列化、锁等待或网络抖动。
108. RAG 改“输入上下文”，微调改“模型参数”。
109. Workflow 是固定流程，Agent 是基于状态/策略做动态决策。
110. 优先测 Tool（确定性高）和 Skill（业务组合逻辑），再做 API 集成测试。
