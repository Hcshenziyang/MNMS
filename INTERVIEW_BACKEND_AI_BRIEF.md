# Project Phoenix 面试讲解稿（后端 + AI 精简版）

## 1. 项目一句话（开场）
这是一个 Agent 化求职辅助系统，后端负责三段闭环：**JD 分析 -> 简历优化 -> 模拟面试**。技术上我重点解决了 **多轮会话编排、RAG 检索、LLM 调用稳定性、容器化部署一致性**。

---

## 2. 后端架构（你主导的部分）

### 2.1 分层设计
- API 层（DRF）：参数校验、鉴权占位、错误响应、事务边界。
- Agent 编排层：`PhoenixAgent` 统一路由到不同 Skill。
- Skill 层：`JDAnalyzer`、`ResumeOptimizer`、`MockInterviewer`。
- Tool 层：LLM 客户端、文本处理、向量检索（FAISS）。
- 数据层：MySQL（会话/消息/分析结果）+ Redis（缓存）+ FAISS 文件索引。

### 2.2 为什么这样拆
核心目标是**可扩展和可测试**：
- View 不堆业务逻辑；
- Skill 专注业务语义；
- Tool 可复用、可替换（比如换模型供应商）。

---

## 3. AI 核心链路（重点讲这个）

### 3.1 JD 分析
1. 接收 JD 文本。
2. 规则抽取关键词（must-have / nice-to-have / 职责）。
3. 组合 Prompt 调用 LLM 输出结构化建议。
4. 结果同时落库（可追溯）并写入会话。

### 3.2 简历优化
1. 读取最近一次 JD 分析结果。
2. 对简历做关键词匹配，计算覆盖度（matched / missing / score）。
3. 调用 LLM 生成“可替换措辞 + 强化建议 + 面试强调点”。

### 3.3 模拟面试（RAG）
1. 题库 Markdown 离线建索引（Embedding -> FAISS）。
2. 按 topic 召回 Top-K 问题。
3. 基于用户回答做点评并追问下一题。
4. 多轮消息写入 `ChatMessage`，形成完整面试轨迹。

## 4. 稳定性与工程化（面试官爱问）
- **缓存策略**：
  - RAG 结果按 topic 缓存（1h）；
  - LLM 结果按 prompt hash 缓存（10min）；
  - 活跃会话最近 10 条消息缓存。
- **降级策略**：
  - 无 API Key 时进入离线回退，系统仍可演示；
  - Embedding 模型不可用时回退到哈希向量，保证检索链路不中断。
- **部署策略**：
  - Docker Compose 编排 MySQL/Redis/Backend/Nginx；
  - 启动脚本自动执行：等待依赖 -> migrate -> build index -> gunicorn。

## 5. 你可以直接说的项目亮点（30 秒）
1. 我把 AI 功能做成了可维护的后端分层，不是把逻辑堆在一个接口里。
2. 我做了 RAG + 缓存 + 降级，保证“有模型更好、没模型也能跑”。
3. 我用容器化把环境一次性打平，开发和演示一致，减少部署不确定性。

## 6. 高频追问与回答模板（精简）

### Q1：为什么不用单个大 Prompt 一把梭？
A：我需要可维护性。单大 Prompt 不利于扩展和排障；分 Skill/Tool 后，每段能力都可独立迭代、测试和替换。

### Q2：RAG 在这里的价值是什么？
A：让模拟面试问题“可控且可追溯”。题目来自可管理题库，不完全依赖模型即兴生成，质量更稳定。

### Q3：如何控制 LLM 成本与延迟？
A：做了 prompt-hash 缓存，重复请求直接命中；同时把确定性任务前置规则抽取，减少无效 token 消耗。

### Q4：系统最重要的可靠性设计是什么？
A：降级机制。包括 LLM 不可用、Embedding 不可用、外部依赖启动慢，都有 fallback，保证主流程不中断。

### Q5：下一步怎么升级？
A：补齐 JWT + 权限、异步任务（Celery）、可观测性（日志/指标/链路）和自动化测试覆盖。
