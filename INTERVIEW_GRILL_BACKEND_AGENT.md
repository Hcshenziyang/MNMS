# 后端 + AI Agent 高压刁难面试题库（基于 Project Phoenix）

> 角色设定：面试官默认不相信“会用框架”，只相信你能解释原理、定位问题、做权衡。
> 使用方式：每题先回答 30 秒，再回答追问 90 秒。

---

## A. 先拷打项目本身（必须答得像你真做过）

1. 你为什么把 Agent 拆成 `API -> Agent -> Skill -> Tool` 四层？如果把逻辑都写在 DRF View 里，具体会在哪三个月后崩掉？
2. 你的 `Skill` 和 `Tool` 边界怎么定的？给一个你项目里“边界模糊但你最终怎么切”的真实例子。
3. 你说系统可扩展。现在要新增“薪资谈判建议”能力，你会改哪些文件？哪些不该改？
4. 你如何证明这个架构不是“为了好看而过度设计”？有没有量化指标支撑？
5. 你现在的 Agent 编排是串行还是并行？为什么？如果并行会引入哪些一致性风险？
6. 你如何保证一轮请求里的 DB 写入和 LLM 调用失败时不会留下脏状态？
7. 你是否做了幂等设计？同一个请求重放 3 次，数据库里会发生什么？
8. 你项目里最容易被面试官说“伪 AI”的点是什么？你怎么反驳？
9. 你说自己做了“闭环”，那闭环的终点是什么数据结构？如何落库并可复用？
10. 你如何定义“这次模拟面试有效”？系统里有任何可验证指标吗？
11. 你现在的 API 返回结构为什么这么设计？哪些字段是给机器用，哪些字段是给人看？
12. 你用了 JSONField，为什么不用严格三范式拆表？你在牺牲什么？
13. 会话模型里 `topic` 放字符串而不是枚举，未来会有什么坑？怎么迁移？
14. 如果用户删了一个会话，你如何保证关联消息、分析结果、缓存、向量引用都清理干净？
15. 你目前的用户模型是简化的。面试官问“这不安全”，你如何在 30 秒内给出升级方案？
16. 你项目里最脆弱的一段代码是哪段？如果线上事故你先怀疑哪一层？
17. 你如何解释“有 LLM Key 和没 LLM Key”两种模式下结果一致性差异？
18. 你的错误码策略是什么？哪些错误返回 4xx，哪些返回 5xx？
19. 如果面试官说“你这只是 ChatGPT 套壳”，你如何用系统设计细节反击？
20. 你能画出一次 `/interview/chat` 请求从 Nginx 到 DB 再到返回的完整时序吗？

## 面试官评分标准（你自测用）

1. 是否能讲出“为什么这样设计”，而不仅是“我用了什么技术”。
2. 是否能给出故障定位路径，而不是泛泛说“看日志”。
3. 是否能在 AI 相关问题里谈工程约束（成本、稳定性、安全、评估）。
4. 是否能承认当前版本不足，并给出可落地升级方案。
5. 是否具备取舍意识：性能、复杂度、交付速度之间怎么平衡。
